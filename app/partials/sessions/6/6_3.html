Bad Parts
Things easy to avoid.
  • == & !=
    Use === and !==
  • with
  • eval
  • continue Statement
    Code can always be refactored to remove it and be more simple.
  • switch Fall Through
    can be error prone
  • Block-less Statements
  • ++ & --
    Most of the buffer overrun bugs that created terrible security vulnerabilities were due to code like this.
  • Bitwise Operators
    In Java, the bitwise operators work with integers. JavaScript doesn’t have integers. It only has double precision floating-point numbers. So, the bitwise operators convert their number operands into integers, do their business, and then convert them back. In most languages, these operators are very close to the hardware and very fast. In JavaScript, they are very far from the hardware and very slow. JavaScript is rarely used for doing bit manipulation. As a result, in JavaScript programs, it is more likely that & is a mistyped && operator. The presence of the bitwise operators reduces some of the language’s redundancy, making it easier for bugs to hide.
  • The function Statement Versus the function Expression
    function statement are always hoisted but expressions are not.
  • Typed Wrappers
    This turns out to be completely unnecessary and occasionally confusing. Don’t use new Boolean or new Number or new String . Also avoid new Object and new Array . Use {} and [] instead.
  • new
    removing new makes any constructor a normal function.
  • void
    In many languages, void is a type that has no values. In JavaScript, void is an opera- tor that takes an operand and returns undefined . This is not useful, and it is very con- fusing. Avoid void .
